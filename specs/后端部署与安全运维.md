# 营销作战地图后端部署与安全运维

## 1. 生产架构

本项目采用 Supabase/PostgreSQL 作为后端：

- Supabase Auth：企业邮箱登录、会话管理、邀请用户。
- PostgreSQL：客户、联系人、项目、活动、提醒、导入任务和审计日志。
- Row Level Security：管理员和售前可见全部数据，销售仅能访问本人负责的数据。
- Edge Function：管理员邀请、停用和启用用户，服务端密钥不会进入浏览器。
- GitHub Pages：仅承载前端静态资源，不保存客户数据。

涉及境内客户数据、联系人信息或敏感项目时，建议在境内云上自托管 Supabase/PostgreSQL，并使用公司域名、HTTPS、WAF、堡垒机和企业备份策略。公开 GitHub Pages 只适合作为演示入口。

## 2. 创建后端

### 2.1 创建项目

创建 Supabase 项目或自托管实例，记录：

- Project URL
- Publishable Key / Anon Key
- Service Role Key（仅用于 Edge Function，禁止放入前端）

关闭公开注册。用户仅允许由管理员邀请。

### 2.2 初始化数据库

在 SQL Editor 执行：

`supabase/migrations/20260629000100_battlemap_production_schema.sql`

该迁移会创建业务表、索引、审计触发器、批量导入函数和 RLS 策略。

### 2.3 创建首位管理员

在 Auth 后台创建首位用户，然后执行：

```sql
update public.profiles
set role = 'admin', display_name = '系统管理员'
where email = '你的管理员邮箱';
```

### 2.4 部署用户管理函数

部署 `supabase/functions/admin-users` 和 `supabase/functions/map-boundary`，并配置：

```text
SITE_URL=https://你的正式访问域名/
ALLOWED_ORIGIN=https://你的正式访问域名
```

`SUPABASE_URL`、`SUPABASE_ANON_KEY`、`SUPABASE_SERVICE_ROLE_KEY` 由函数运行环境提供。Service Role Key 不得写入 Git、前端环境变量或浏览器。

## 3. 配置前端

复制环境变量模板：

```bash
cp .env.example .env.production
```

填写：

```text
VITE_SUPABASE_URL=https://your-project.supabase.co
VITE_SUPABASE_ANON_KEY=your-publishable-key
```

构建：

```bash
npm ci
npm run build:production
```

未配置这两个参数时，系统会明确显示“演示数据模式”，数据只保存在当前浏览器；配置后将强制显示登录页并使用后端数据。

## 4. 权限验证清单

- 销售账号只能读取、创建、更新本人负责的客户、联系人和项目。
- 销售不能把项目负责人改为其他人。
- 售前可以读取全部业务数据，但不能进入系统管理。
- 管理员可以读取全部数据、管理用户和查看审计日志。
- 未登录请求无法读取任何业务表。
- 浏览器端仅配置 Publishable/Anon Key，不配置 Service Role Key。

## 5. 上线前必做

- 配置企业 SMTP，确保邀请和重置密码邮件可达。
- 开启 MFA，管理员账号优先强制启用。
- 配置每日备份；重要环境使用时间点恢复。
- 配置错误监控、接口延迟监控、数据库容量告警和登录异常告警。
- 对联系人手机号、邮箱和微信号设为机密数据，并制定导出审批规则。
- 每季度复核用户、角色、离职账号、共享账号和服务端密钥。
- 不在公开演示环境录入真实客户数据。

## 6. 地图数据

全国省级边界随前端发布；省、市、区县边界按需从阿里云 DataV 行政区划服务加载。正式环境应通过企业网关代理并缓存，记录边界数据版本，避免外部服务异常影响地图下钻。
